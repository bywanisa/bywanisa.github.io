Experimental: Thai LegalCraft

Transforming Legal Complexity into Structured, Self-Service Automation

Snapshot

Role: Product Owner (POC), UX Designer, Workflow Architect

Timeline: 3 days

Outcome: Working end-to-end prototype generating legally formatted, print-ready agreements from structured inputs.

Primary Focus: MVP definition, requirement framing, automation feasibility, AI-accelerated delivery.

Problem

Creating legal agreements in Thailand is often slow, costly, and inaccessible to people without legal knowledge. Most individuals rely on unreliable templates or expensive consultations for relatively standard contracts.

Opportunity

Design a self-service system that converts plain user inputs into properly formatted legal documents in minutes.

Target Users

  • Individuals issuing or receiving private loans
  • Small business owners
  • Users without legal or document-formatting expertise

Success Criteria (POC Level)

  • A user can complete required information without legal knowledge
  • System outputs a structured, print-ready PDF
  • End-to-end flow works reliably
  • Deliver within a 72-hour build window

My Role & Responsibilities

In this sprint I operated across multiple functions:

  • Define product scope & MVP boundary
  • Translate legal structure → system logic
  • Design input flow & live document preview
  • Map data contracts between UI → workflow → template engine
  • Prioritize speed to validation over visual perfection
  • Ship a deployable prototype

Approach

I. Product Framing & MVP Strategy

Given the extreme time constraint, I prioritized:

  • Core legal completeness
  • Deterministic output
  • Automation reliability

Instead of UI polish or scalability. I intentionally framed this as a validation vehicle, not a production system.

II. Experience Design

The primary UX challenge was reducing cognitive load from legal terminology.

Key principles:
  • Plain-language inputs
  • Structured step flow
  • Real-time A4 preview to build user confidence
  • Smart defaults and conditional visibility
  • Printable output with manual fill capability for edge cases
III. System & Workflow Design

I designed the orchestration between:

Form Input → JSON Payload → n8n Workflow → Google Docs Template → PDF → Browser Download

Focus areas:
  • Data integrity
  • Authentication
  • Transformation reliability
  • Handling binary responses back to the client
IV. AI as Delivery Accelerator

Using an AI-native IDE shifted my effort toward:

  • Requirement clarity
  • Prompt precision
  • Validation of outputs
  • Iteration speed

This mirrors how modern product teams increasingly operate.

Key Decisions & Trade-offs

To meet the deadline:

  • Chose low-code orchestration over custom services
  • Accepted template limitations in exchange for rapid validation
  • Deferred advanced UI states and multi-document management

If scaling for volume, I would migrate toward a dedicated rendering pipeline.

Challenges Solved

  • Converting structured financial inputs into legal language formats
  • Thai currency text including Satang handling
  • Returning binary PDF streams securely to the UI
  • Designing fallbacks for optional or unknown information

Impact

Within 72 hours, the concept moved from idea → usable product simulation.

This enabled:

  • Demonstration of feasibility
  • Stakeholder conversation around automation
  • A base architecture for expansion

What This Demonstrates

  • Ability to define and protect MVP
  • Translating business rules into system behavior
  • Cross-functional execution (UX / BA / PO mindset)
  • Leveraging AI to shorten delivery cycles

Future Direction

If continued, the next steps would be:

  • Demonstration of feasibility

Reflections

This project reinforced a principle I strongly believe in
Speed to validated learning often creates more value than premature perfection.

Trust is the real problem

Legal automation is not only technical. Users struggle because they:

  • Do not understand legal terminology
  • Fear making mistakes
  • Doubt whether the output is truly usable

Elements like plain-language inputs and real-time previews therefore served one purpose, build confidence.

Constraint creates discipline

With only 72 hours, strong scope control was mandatory. I intentionally excluded:

  • Do not understand legal terminology
  • Fear making mistakes
  • Doubt whether the output is truly usable

These are important but they are post-validation investments.

Outcome over completeness

For this POC, my definition of done was simple
If a user can independently generate a valid printable agreement, the system succeeds.

Anything beyond that would reduce focus on learning.

Designed to prove, not to scale

The architecture supports feasibility demonstration and stakeholder discussion. It is not intended for high-volume production.

A real rollout would require further work in rendering scalability, governance, compliance ownership, and lifecycle operations.

AI accelerated execution, not responsibility

AI increased build speed. However, value only appeared when:

  • Requirements were explicit
  • Constraints were clear
  • Outputs were critically evaluated

Problem framing and decision accountability remained human.

Personal shift toward product thinking

I became more comfortable releasing with imperfections as long as the release produced:

  • Evidence
  • Momentum
  • Clearer next priorities

Progress is measured by validated direction, not feature completeness.

The real achievement was not generating PDFs. It was turning ambiguity into a reliable, testable outcome within extreme constraints.
